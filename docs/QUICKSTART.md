# Spirit 精灵 - 快速开始指南

## 问题背景

你遇到的问题：
1. **LLM 没有读取 PR 评论** - 提交代码后，reviewer 在 GitLab MR 上留下评论，但 LLM 没有自动读取并修复
2. **代码上下文不足** - LLM 修复时没有读取足够的代码上下文，导致修复不完整
3. **缺少本地验证** - 没有在提交前执行 `go vet`、`go build` 等验证，导致流水线报错

## 解决方案

Spirit 现在提供 **MR 评论自动修复** 功能，完整解决上述问题。

## 快速开始

### 1. 配置 Spirit

编辑 `config.local.yaml`：

```yaml
environments:
  - name: production
    type: prod
    elasticsearch:
      url: "http://your-es-server:9200"
      indices: ["app-prod-*"]
      errorQuery: "level:ERROR OR level:FATAL"
    logFiles: []
    databases: []
    codeRoots:
      - name: backend
        path: "/Users/alex/test/codebase/mobgi_ai_backend"
        gitRemote: "origin"
        defaultBranch: "master"
        gitlabProjectPath: "idreamsky/advertisement/PlatformProducts/mobgi_ai_backend"

gitlab:
  url: "https://gitlab.example.com"
  token: "glpat-xxxxxxxxxxxx"  # 你的 GitLab Token
  defaultTargetBranch: "test"

claude:
  apiKey: "sk-xxxxxxxxxxxx"  # 你的 Claude API Key
  baseUrl: "https://api.anthropic.com/v1"
  model: "claude-sonnet-4-20250514"

watcher:
  esPollingInterval: 30
  dedupeWindow: 3600
  maxConcurrentFixes: 3
  riskAutoFix: ["A", "B"]
  mrCommentPollingInterval: 60  # MR 评论轮询间隔（秒）

storage:
  type: "sqlite"
  path: "./data/incidents.db"
```

### 2. 启动 Spirit Watcher

```bash
cd /Users/alex/test/shell/spirit
npm run watcher
```

你会看到类似输出：

```

Environments: production
ES polling interval: 30s
Dedupe window: 3600s
Max concurrent fixes: 3
Auto-fix risk levels: A, B
Target branch: test
MR comment polling: 60s
MR Comment Poller starting (interval: 60s)
=== Spirit 精灵 Watcher running ===
```

### 3. 测试 MR 评论检测

在另一个终端运行测试脚本：

```bash
cd /Users/alex/test/shell/spirit
npx tsx src/test-mr-comment-poller.ts
```

这会检查所有现有 MR 的评论状态。

### 4. 实际使用场景

#### 场景：修复 unused import 错误

**步骤 1：Spirit 自动创建 MR**

Spirit 检测到错误日志 → 分析 → 自动修复 → 创建 MR !123

**步骤 2：Reviewer 发现问题并评论**

在 GitLab MR !123 上评论：

```
internal/workerjobs/job_video.go:11:2 这行导入了 
"mobgi_ai_backend/internal/integration" 但没有使用，
导致 go vet 报错。请删除这个 unused import。
```

**步骤 3：Spirit 自动处理**

60 秒内（默认轮询间隔），Spirit 会：

1. ✅ 检测到新评论
2. ✅ 读取 `internal/workerjobs/job_video.go` 完整内容
3. ✅ 调用 Claude 分析评论并生成修复方案
4. ✅ 在 worktree 中应用修复（删除 unused import）
5. ✅ 执行 `go vet ./...` 验证
6. ✅ 执行 `go build ./...` 验证
7. ✅ 提交：`fix(spirit): address review feedback`
8. ✅ 推送到原分支

**步骤 4：Reviewer 验证**

刷新 MR 页面，看到新的 commit，验证问题已修复 → 批准合并

### 手动触发测试

如果想立即测试某个 MR 的评论处理，可以：

1. 在 MR 上添加一条测试评论
2. 等待最多 60 秒（或你配置的轮询间隔）

```bash

npm run watcher
```

## 常见问题

### Q1: Spirit 没有响应我的评论

**检查清单：**

1. Watcher 是否在运行？
   ```bash
   ps aux | grep watcher
   ```

2. 评论是否被识别为人工评论？
   - 不能以 `## 🤖` 或 `## 🔧` 开头
   - 不能包含 "Auto-generated by Spirit"

3. GitLab Token 是否有效？
   ```bash
   curl -H "PRIVATE-TOKEN: your-token" \
     https://gitlab.example.com/api/v4/user
   ```

4. 查看日志是否有错误：
   ```bash
   tail -50 spirit-watcher.log
   ```

### Q2: 修复后仍然有构建错误

**可能原因：**
- Claude 理解有偏差
- 代码上下文不够完整
- 依赖关系复杂

**解决方法：**
1. 在评论中提供更详细的说明，包括：
   - 具体文件名和行号
   - 期望的修改内容
   - 相关的依赖关系

2. 示例好评论：
   ```
   文件 internal/workerjobs/job_video.go 第 11 行：
   
   当前代码：
   import "mobgi_ai_backend/internal/integration"
   
   问题：这个 import 没有被使用
   
   修复：删除这一行 import
   ```

### Q3: 想要更快的响应速度

**调整轮询间隔：**

编辑 `config.local.yaml`：

```yaml
watcher:
  mrCommentPollingInterval: 30  # 改为 30 秒
```

重启 Watcher 生效。

**注意：** 间隔太短会增加 GitLab API 调用频率，建议不要低于 30 秒。

### Q4: 如何跳过某些评论

在评论中添加 `[skip-spirit]` 标记：

```
[skip-spirit] 这个问题需要人工处理，涉及复杂的业务逻辑调整。
```

Spirit 会忽略包含此标记的评论。

## 进阶配置

### 针对不同项目类型的验证

Spirit 会自动检测项目类型并执行相应的验证：

**Go 项目：**
- 检测 `go.mod` 文件
- 执行 `go vet ./...`
- 执行 `go build ./...`

**Node 项目：**
- 检测 `package.json` 文件
- 执行 `npx tsc --noEmit`

### 自定义验证命令

如果需要自定义验证命令，可以修改 `src/watcher/auto-fix.ts` 中的 `runLocalVerification` 函数。
